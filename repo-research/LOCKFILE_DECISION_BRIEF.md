## 2. Why Dual Lockfiles Are A Risk

- **Install-time drift.** If a contributor runs `pnpm install` while the canonical lockfile is `package-lock.json`, dependencies are resolved from pnpm's resolver, the resulting `node_modules` may differ from what `package-lock.json` describes, and CI parity is broken.
- **CI confusion.** CI cannot tell which lockfile to use without a config file (`.npmrc`, `.pnpmrc`, `packageManager`, or a script that asserts which one is present).
- **Repo bloat.** Two lockfiles double the diff surface for dependency changes.
- **Review noise.** A PR that updates only one lockfile becomes ambiguous — is the other intentionally out of date?
- **Future tool compatibility.** Some tools assume exactly one lockfile format.

## 3. npm Option

**Pros:**

- `package-lock.json` is the most common format; lowest friction for new contributors.
- Matches `package.json` scripts (no shell differences).
- Cloudflare Pages default build command (`npm run build`) works unchanged.
- `docs/SETUP.md` and `docs/DEPLOYMENT.md` already implicitly assume npm.

**Cons:**

- npm's resolver is less strict than pnpm's; small reproducibility risk vs pnpm.
- npm does not enforce the same hoisting rules.

**Migration cost (if pnpm is dropped):**

1. Delete `pnpm-lock.yaml`.
2. Verify `package-lock.json` is up to date with current `package.json` (`npm install`).
3. Update `docs/SETUP.md`, `docs/DEPLOYMENT.md` to drop pnpm mentions.
4. Update `README.md` (per README repair spec) to recommend npm.
5. Add a CI assertion (in a tooling-approved phase) that fails the build if `pnpm-lock.yaml` reappears.

## 4. pnpm Option

**Pros:**

- Stricter resolver; better reproducibility.
- Faster installs in CI; smaller `node_modules` due to symlinking.
- `pnpm-lock.yaml` is lockfileVersion 9, indicating a modern pnpm.

**Cons:**

- Smaller contributor pool familiar with pnpm.
- Some tools assume npm by default; more setup work.
- `package.json` scripts do not need changes but `docs/SETUP.md` and `docs/DEPLOYMENT.md` would.
- Cloudflare Pages build command would change to `pnpm run build`.

**Migration cost (if npm is dropped):**

1. Delete `package-lock.json`.
2. Verify `pnpm-lock.yaml` is up to date with current `package.json` (`pnpm install`).
3. Update `docs/SETUP.md`, `docs/DEPLOYMENT.md` to use pnpm.
4. Update `README.md` (per README repair spec) to recommend pnpm.
5. Add `packageManager: "pnpm@<version>"` to `package.json` so Corepack / npm / pnpm all agree.
6. Add a CI assertion that fails the build if `package-lock.json` reappears.

## 5. Recommended Choice

**Recommendation: keep npm.** Reasons:

- `package-lock.json` is present and current.
- `package.json` scripts and `docs/DEPLOYMENT.md` already assume npm.
- `docs/SETUP.md` already recommends npm until the owner decides.
- Lower contributor friction.
- No `package.json` change required.

This is a recommendation, not a decision. The owner must confirm.
